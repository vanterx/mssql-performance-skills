## Category 1 — Login Portability (J1–J5)

### J1 — Orphaned Database User After Restore

**What it means:** A database is restored or seeded onto a new instance, but the logins are
created separately (or already existed) with a different SID than the database user expects.
SQL Server maps users to logins by SID, not by name — a name match with a SID mismatch is still
an orphan.

**How to spot it:**
```sql
SELECT dp.name AS user_name
FROM sys.database_principals dp
LEFT JOIN sys.server_principals sp ON dp.sid = sp.sid
WHERE dp.type IN ('S','U') AND sp.sid IS NULL AND dp.principal_id > 4;
```

**Example:**
```sql
-- Problem: AppUser exists in the database but has no matching login SID on the target
ALTER USER [AppUser] WITH LOGIN = [AppUser];
-- Run only after a login named AppUser with the correct SID exists on the target
```

**Fix options:**
1. Prevent the problem at the source: always generate target login-creation scripts with
   `sp_help_revlogin` so SIDs match exactly (see J2).
2. If orphaned users already exist, run `ALTER USER ... WITH LOGIN` to re-map each one to its
   correct login.
3. Re-check role membership after re-mapping — orphan repair alone does not restore database role
   membership (see J7).

**Related checks:** J2 (root cause prevention), J7 (role membership lost alongside orphaning)

### J2 — SQL Authentication Login SID Mismatch Risk

**What it means:** Manually typing `CREATE LOGIN [name] WITH PASSWORD = '...'` generates a brand
new SID every time, even if the name matches exactly. This is the most common root cause of J1.
The native, SID-preserving alternative is the documented `sp_help_revlogin` script.

**How to spot it:** Migration plan or script set creates SQL logins with `CREATE LOGIN`
statements that specify a password rather than a hashed password and SID.

**Example:**
```sql
-- Problem (causes new SID, will orphan every database user):
CREATE LOGIN [AppUser] WITH PASSWORD = 'TempPass123!';

-- Fix (SID-preserving, generated by sp_help_revlogin on the source):
CREATE LOGIN [AppUser] WITH PASSWORD = 0x0200F2A1... HASHED,
  SID = 0x88E1A7C9F3B24..., DEFAULT_DATABASE = [SalesDB], CHECK_POLICY = OFF;
```

**Fix options:**
1. Run `sp_help_revlogin` on the source (in-box system stored procedure, documented by
   Microsoft) and execute its generated script on the target verbatim.
2. Never hand-type `CREATE LOGIN ... WITH PASSWORD` for a migration where the source login
   already exists — that path is only correct for brand-new logins that have no pre-existing
   database user mappings.

**Related checks:** J1 (the symptom this check prevents)

### J3 — Login Type Unsupported on Target Platform

**What it means:** Not every login type exists everywhere. Windows-authenticated logins have no
equivalent on Azure SQL Database; certificate-mapped and asymmetric-key-mapped logins may also
be unsupported depending on the specific target platform tier.

**How to spot it:**
```sql
SELECT name, type_desc FROM sys.server_principals
WHERE type_desc IN ('WINDOWS_LOGIN','WINDOWS_GROUP','CERTIFICATE_MAPPED_LOGIN','ASYMMETRIC_KEY_MAPPED_LOGIN');
```

**Example:**
```
-- Problem: DOMAIN\svc_app is WINDOWS_LOGIN, target is Azure SQL Database
-- Fix: create a Microsoft Entra ID-backed user instead
CREATE USER [svc_app@yourtenant.onmicrosoft.com] FROM EXTERNAL PROVIDER;
```

**Fix options:**
1. Map each unsupported login type to its target-supported equivalent before migrating — Entra
   ID for Windows auth on Azure SQL Database, contained SQL auth users as a fallback.
2. If the target is Azure SQL Managed Instance instead, Windows auth via Entra ID-joined AD DS
   is supported with different setup steps — confirm which Azure tier is actually the target.

**Related checks:** none in this skill — see `/sqlmigration-review` Y8 for the platform-level
trigger that this check details the login-by-login remediation for.

### J4 — Password Policy / Expiration Settings Differ From Source

**What it means:** `CHECK_POLICY`/`CHECK_EXPIRATION` on a SQL login defer to the Windows domain's
password policy when set ON. If the target server is not domain-joined the same way (or is
Azure SQL Database, which has its own independent policy model), the inherited behavior changes
silently — logins that were enforced may become unenforced, or vice versa.

**How to spot it:**
```sql
SELECT name, is_policy_checked, is_expiration_checked
FROM sys.sql_logins;
```

**Example:**
```
-- Problem: AppUser has CHECK_POLICY = ON; target is a workgroup server with no domain policy
-- Fix: explicitly decide the policy rather than relying on inherited domain behavior
ALTER LOGIN [AppUser] WITH CHECK_POLICY = ON; -- re-state explicitly after confirming local policy
```

**Fix options:**
1. Re-evaluate `CHECK_POLICY`/`CHECK_EXPIRATION` for each login explicitly on the target rather
   than assuming the setting's effective behavior carries over.
2. Document the decision for security review sign-off, since password policy is often an audit
   requirement.

**Related checks:** J3 (login type/platform differences generally)

### J5 — Migrated Login's Default Database Does Not Exist on Target

**What it means:** A login's `default_database_name` is the database the connection lands in
when no database is specified in the connection string. If that database wasn't part of the
migration scope, or was renamed, login succeeds but `USE` fails or the application lands in the
wrong context.

**How to spot it:**
```sql
SELECT sp.name, sp.default_database_name
FROM sys.server_principals sp
WHERE sp.default_database_name NOT IN (SELECT name FROM sys.databases);
```

**Example:**
```sql
ALTER LOGIN [AppUser] WITH DEFAULT_DATABASE = [SalesDB_New];
```

**Fix options:**
1. Update `DEFAULT_DATABASE` for every affected login once the target database name is final.
2. Cross-check this against the migration scope list — if a login's default database isn't being
   migrated at all, confirm that's intentional.

**Related checks:** none — standalone connectivity check.

## Category 2 — Permission Fidelity (J6–J9)

### J6 — Server-Level Role Membership Not Re-Created

**What it means:** Fixed server roles (`sysadmin`, `dbcreator`, etc.) and user-defined server
roles are not part of a login's `CREATE LOGIN` statement — they are a separate membership grant
that must be scripted independently, and `sp_help_revlogin` does not capture this.

**How to spot it:**
```sql
SELECT sp.name AS login_name, r.name AS role_name
FROM sys.server_role_members rm
JOIN sys.server_principals sp ON rm.member_principal_id = sp.principal_id
JOIN sys.server_principals r ON rm.role_principal_id = r.principal_id;
```

**Example:**
```sql
ALTER SERVER ROLE [dbcreator] ADD MEMBER [DBA_Login];
```

**Fix options:**
1. Capture `sys.server_role_members` on the source before migrating, and script the
   `ALTER SERVER ROLE ... ADD MEMBER` statements to run immediately after login creation.
2. For user-defined server roles, also script the `CREATE SERVER ROLE` statement first if the
   role itself doesn't exist on the target.

**Related checks:** J7 (the database-level equivalent of this same gap)

---

### J7 — Database Role Membership Lost on User Recreation

**What it means:** Fixing an orphaned user (J1) re-maps the user to a login but does not restore
that user's membership in database roles (`db_datareader`, custom roles, etc.) — role membership
is tracked separately and is not implied by the user-login mapping.

**How to spot it:**
```sql
SELECT dp.name AS user_name, r.name AS role_name
FROM sys.database_role_members rm
JOIN sys.database_principals dp ON rm.member_principal_id = dp.principal_id
JOIN sys.database_principals r ON rm.role_principal_id = r.principal_id;
```

**Example:**
```sql
ALTER ROLE [db_datareader] ADD MEMBER [AppUser];
```

**Fix options:**
1. Capture `sys.database_role_members` before migrating, alongside the login/user capture.
2. Re-run the role membership grants immediately after orphan repair, not as an afterthought —
   a user with a working login but missing role membership produces confusing "permission
   denied" errors that look unrelated to the migration.

**Related checks:** J1, J6

---

### J8 — Explicit Object-Level GRANT/DENY Not Captured in Migration Scripts

**What it means:** Many migration scripts only handle logins, users, and role membership —
explicit object-level permissions (a `GRANT SELECT` on one specific table, a `DENY` overriding a
role grant) are easy to miss because they aren't visible from the login/role view alone.

**How to spot it:**
```sql
SELECT pr.name AS grantee, pe.permission_name, pe.state_desc, o.name AS object_name
FROM sys.database_permissions pe
JOIN sys.database_principals pr ON pe.grantee_principal_id = pr.principal_id
LEFT JOIN sys.objects o ON pe.major_id = o.object_id
WHERE pe.major_id > 0;
```

**Example:**
```sql
-- Problem: AppReporting has an explicit DENY on dbo.SalaryDetails overriding its db_datareader
--          grant; the migration script only handled db_datareader role membership
DENY SELECT ON dbo.SalaryDetails TO [AppReporting];
```

**Fix options:**
1. Script `sys.database_permissions` explicitly before migrating, not just role membership.
2. Apply explicit GRANT/DENY statements after the role-based grants, since DENY overrides GRANT
   regardless of order in SQL Server's actual permission evaluation — but script them as a
   distinct, reviewed step so they aren't silently dropped.

**Related checks:** J6, J7 (role-based permission migration, which this check supplements)

### J9 — Cross-Database Ownership Chain Breaks at Migration Boundary

**What it means:** Ownership chaining lets a stored procedure in DatabaseA implicitly access
objects in DatabaseB if both objects share the same owner and chaining is enabled — but this
depends on both databases existing together with matching ownership and `TRUSTWORTHY`/chaining
settings. A partial migration (one database moves, the other doesn't, or settings differ) breaks
the chain.

**How to spot it:**
```sql
SELECT referencing_entity_name, referenced_database_name, referenced_entity_name
FROM sys.sql_expression_dependencies
WHERE referenced_database_name IS NOT NULL
  AND referenced_database_name != DB_NAME();
```

**Example:**
```
-- Problem: dbo.GetCustomerOrders in SalesDB references OrdersDB.dbo.Orders by ownership
--          chaining; only SalesDB is in the migration's first wave
-- Fix: migrate both databases together, or replace the chain with an explicit grant
```

**Fix options:**
1. Migrate dependent databases together in the same wave whenever ownership chaining is
   detected.
2. If they cannot move together, replace the implicit chain with an explicit cross-database
   permission grant secured by a module-signing certificate — never enable `TRUSTWORTHY ON` as a
   substitute, since it grants broader implicit trust than the original chain required.

**Related checks:** none in this skill — see `/sqlmigration-objects-review` for non-ownership
cross-database dependencies like linked servers.

## Category 3 — Credentials & Secrets (J10–J12)

### J10 — SQL Server Credential Object Not Migrated

**What it means:** `sys.credentials` rows back proxy accounts, CLR external access, and external
data sources. SQL Server deliberately does not expose the stored secret through any system view
or scripting tool — there is no `sp_help_revlogin`-style equivalent for credentials.

**How to spot it:**
```sql
SELECT name, credential_identity FROM sys.credentials;
```

**Example:**
```sql
CREATE CREDENTIAL [AzureBlobCred]
  WITH IDENTITY = 'SHARED ACCESS SIGNATURE',
  SECRET = '<sas-token-from-original-secret-store>';
```

**Fix options:**
1. Retrieve each credential's secret from its original source (a secrets vault, deployment
   pipeline variable, or documented runbook) — not from the SQL Server instance, which cannot
   provide it.
2. Recreate the credential on the target with the same name so dependent proxy/job step
   references resolve without modification.

**Related checks:** J11 (proxy accounts that reference credentials by ID)

---

### J11 — SQL Agent Proxy Account References a Credential Not Yet Created

**What it means:** A proxy account (`msdb.dbo.sysproxies`) stores a `credential_id` foreign key.
If the migration script creates the proxy before the credential exists, or maps the wrong
`credential_id` (IDs are not guaranteed identical across instances), the proxy is broken even
though it appears to exist.

**How to spot it:**
```sql
SELECT p.name AS proxy_name, c.name AS credential_name
FROM msdb.dbo.sysproxies p
JOIN sys.credentials c ON p.credential_id = c.credential_id;
```

**Example:**
```sql
-- Sequence: credential first, then proxy referencing it by name lookup, not by hardcoded ID
DECLARE @cred_id INT = (SELECT credential_id FROM sys.credentials WHERE name = 'AzureBlobCred');
EXEC msdb.dbo.sp_add_proxy @proxy_name = 'BlobProxy', @credential_id = @cred_id;
```

**Fix options:**
1. Sequence credential creation strictly before proxy creation in the migration script.
2. Always resolve `credential_id` by name lookup in the target's own `sys.credentials`, never by
   copying the source's numeric ID.

**Related checks:** J10

---

### J12 — Linked Server Stored Login Not Migrated

**What it means:** `sys.linked_logins` maps a local login to a remote login/password (or
impersonation) used when a linked server query executes. Like credentials, the stored remote
password is never exposed by any system view.

**How to spot it:**
```sql
SELECT ll.name AS local_login, s.name AS linked_server, ll.uses_self_credential
FROM sys.linked_logins ll
JOIN sys.servers s ON ll.server_id = s.server_id;
```

**Example:**
```sql
EXEC sp_addlinkedsrvlogin 'RemoteServer', 'false', 'LocalLogin', 'RemoteLogin', 'RemotePassword';
```

**Fix options:**
1. Re-enter the remote login's password manually on the target — retrieve it from the original
   secret store, the same as J10.
2. Confirm `uses_self_credential` mappings (impersonation) are reconfirmed against the target's
   own login set rather than assumed to carry over by name.

**Related checks:** J10 — see `/sqlmigration-objects-review` for the rest of the linked server
object definition (data source, provider, options).

## Category 4 — Certificates & Keys (J13–J14)

### J13 — Certificate-Backed Object Migrated Without Its Certificate

**What it means:** Certificate-mapped logins, Service Broker security, and module-signing
certificates are separate objects from the things they protect — migrating the protected object
(a login, a queue, a signed procedure) without also migrating the certificate it depends on
leaves that dependency unsatisfiable on the target.

**How to spot it:**
```sql
SELECT name, expiry_date, pvt_key_encryption_type_desc FROM sys.certificates
WHERE is_active_for_begin_dialog = 0; -- excludes TDE-only certs typically flagged elsewhere
```

**Example:**
```sql
-- On source:
BACKUP CERTIFICATE [SBCert] TO FILE = 'C:\Certs\SBCert.cer'
  WITH PRIVATE KEY (FILE = 'C:\Certs\SBCert.pvk', ENCRYPTION BY PASSWORD = 'P@ssw0rd!');
-- On target, before dependent objects are created:
CREATE CERTIFICATE [SBCert] FROM FILE = 'C:\Certs\SBCert.cer'
  WITH PRIVATE KEY (FILE = 'C:\Certs\SBCert.pvk', DECRYPTION BY PASSWORD = 'P@ssw0rd!');
```

**Fix options:**
1. Inventory all non-TDE certificates and their dependents before migrating.
2. Back up and restore each certificate (with private key, if it has one) before creating the
   objects that depend on it — sequencing matters, since `CREATE CERTIFICATE` for the dependent
   object's prerequisite must complete first.

**Related checks:** J14 — see `/sqlencryption-review` for TDE-specific certificate migration.

---

### J14 — Database Master Key Not Backed Up Before Migration

**What it means:** The Database Master Key (DMK) protects certificates and asymmetric keys
within the database. If it isn't explicitly backed up and restored as part of the migration, any
encryption hierarchy built on top of it (Always Encrypted CMKs, column-level encryption keys)
becomes inaccessible after the database moves, even though the database itself restores fine.

**How to spot it:**
```sql
SELECT name FROM sys.symmetric_keys WHERE name = '##MS_DatabaseMasterKey##';
```

**Example:**
```sql
-- On source:
OPEN MASTER KEY DECRYPTION BY PASSWORD = '<source-password>';
BACKUP MASTER KEY TO FILE = 'C:\Keys\dmk.key' ENCRYPTION BY PASSWORD = '<export-password>';
-- On target:
RESTORE MASTER KEY FROM FILE = 'C:\Keys\dmk.key'
  DECRYPTION BY PASSWORD = '<export-password>' ENCRYPTION BY PASSWORD = '<target-password>';
```

**Fix options:**
1. Always include a DMK backup/restore step in the migration plan whenever
   `##MS_DatabaseMasterKey##` is present, regardless of whether encryption usage is otherwise
   obvious.
2. See `/sqlencryption-review` for the full DMK/SMK key hierarchy migration sequence if the
   database uses Always Encrypted, column-level encryption, or `sp_control_dbmasterkey_password`
   auto-open.

**Related checks:** J13

## Category 5 — CMS (J15)

### J15 — Central Management Server Registration Points at Old Instance Name

**What it means:** Central Management Server registrations are a convenience/inventory layer on
top of the actual SQL Server instances — they are stored centrally (`msdb` on the CMS host) and
do not update automatically when an instance is renamed or replaced as part of a migration.

**How to spot it:** Compare CMS registered server names/connection strings against the
post-migration instance name or listener name.

**Example:**
```
-- Old registration points at SQLPROD01; post-migration the application connects via
-- prod-ag-l.corp.local (an AG listener) — the CMS entry is now stale for monitoring purposes
```

**Fix options:**
1. Update the registration via SSMS Registered Servers (Object Explorer right-click → Properties)
   after cutover is confirmed stable.
2. For bulk updates, script against
   `msdb.dbo.sp_sysmanagement_update_shared_registered_server` on the CMS instance.

**Related checks:** none — this is a monitoring/inventory hygiene check, not a functional
migration blocker.
